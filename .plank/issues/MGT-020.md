# MGT-020: ScalarResult `__iter__` / `__next__` iteration

## Description

In SA v2, `ScalarResult` is iterable — you can write `for user in result:`.
Currently mongotic's `ScalarResult` only supports `.all()`, `.first()`, etc.
Add iterator protocol support.

## Expected API

```python
result = session.scalars(select(User).where(User.active == True))

# Direct iteration
for user in result:
    print(user.name)

# Also works with list comprehension
names = [u.name for u in session.scalars(stmt)]

# Unpacking
first, second, *rest = session.scalars(stmt)
```

## Implementation Guidelines

1. **Add `__iter__` to `ScalarResult`** that builds the cursor and yields
   hydrated model instances one at a time.
2. **Add `__next__`** by making `ScalarResult` a proper iterator, or return
   `self` from `__iter__` and track cursor state.
3. **Preferred approach**: implement `__iter__` as a generator that calls
   `_build_cursor()` and yields `_hydrate(doc)` for each document. This is
   simpler and avoids managing cursor state manually.
4. **Ensure `.all()`, `.first()`, etc. still work** — they should remain
   independent of the iterator state (each builds its own cursor).

### Key design considerations

- Each call to `__iter__` should create a fresh cursor to allow multiple
  iterations over the same `ScalarResult`.
- Be mindful of cursor exhaustion — once iterated, a cursor-based iterator
  cannot be restarted. Using a fresh cursor per `__iter__` call avoids this.
- Type hint: `ScalarResult[_T]` should satisfy `Iterable[_T]`.

## Acceptance Criteria

- [ ] `for item in result:` works
- [ ] Multiple iterations over same `ScalarResult` produce same results
- [ ] Type inference works: iterating `ScalarResult[User]` yields `User`
- [ ] Existing `.all()` / `.first()` etc. are unaffected
- [ ] Tests in `tests/test_scalar_result_iter.py`
