# MGT-019: Null operators — `.is_(None)` / `.is_not(None)`

## Description

MongoDB stores `null` values and supports querying for them. Add `.is_()`
and `.is_not()` methods to `ModelField` following the SA v2 convention.

## Expected API

```python
# Find users with no email set
stmt = select(User).where(User.email.is_(None))

# Find users that have an email
stmt = select(User).where(User.email.is_not(None))
```

## Implementation Guidelines

1. **Add `is_(self, value)` to `ModelField`**:
   - When `value is None` → return `ModelFieldOperation` with a new `IS_NULL`
     operator (or reuse `EQUAL` with `None` value since MongoDB `{"field": None}`
     matches null / missing).
   - Optionally support non-None values as well (equivalent to `==`).
2. **Add `is_not(self, value)` to `ModelField`**:
   - When `value is None` → return `ModelFieldOperation` with `NOT_EQUAL` and
     value `None`, which maps to `{"field": {"$ne": null}}`.
   - Alternatively, use `not_()` from MGT-018 internally.
3. **Update `to_mongo_filter()`** to handle `None` values correctly — MongoDB
   natively supports `{"field": None}` to match null/missing fields.

### Key design considerations

- In MongoDB, `{"field": None}` matches documents where the field is `null`
  **or** the field does not exist. Document this behaviour.
- `is_not(None)` should return documents where the field exists **and** is not
  null — `{"field": {"$ne": None}}` achieves this.
- Keep API naming identical to SA v2: `is_()` and `is_not()` (with trailing
  underscore to avoid Python keyword conflict).

## Acceptance Criteria

- [ ] `ModelField.is_(None)` produces `{"field": None}` filter
- [ ] `ModelField.is_not(None)` produces `{"field": {"$ne": None}}` filter
- [ ] Composes correctly with `or_()`, `and_()`, `not_()`
- [ ] Tests in `tests/test_null_operators.py`
