# MGT-018: Logical combinators — `or_()`, `and_()`, `not_()`

## Description

Currently `.where()` only supports implicit AND (multiple `ModelFieldOperation`
arguments are AND-ed together). There is no way to express OR or NOT conditions,
which severely limits query expressiveness.

Add top-level `or_()`, `and_()`, `not_()` functions that produce composable
filter nodes, mapping to MongoDB's `$or`, `$and`, `$not` / `$nor` operators.

## Expected API

```python
from mongotic import select, or_, and_, not_

# OR
stmt = select(User).where(
    or_(User.role == "admin", User.role == "moderator")
)

# NOT
stmt = select(User).where(
    not_(User.status == "banned")
)

# Nested composition
stmt = select(User).where(
    and_(
        User.age >= 18,
        or_(User.role == "admin", User.verified == True),
    )
)

# NOT with complex expression
stmt = select(User).where(
    not_(or_(User.role == "guest", User.role == "anonymous"))
)
```

## Implementation Guidelines

1. **Define a `CompoundFilter` class** (or similar) that wraps a list of
   `ModelFieldOperation | CompoundFilter` with an operator (`$and` / `$or` / `$nor`).
2. **`or_(*conditions)`** → returns `CompoundFilter(op="$or", children=[...])`.
3. **`and_(*conditions)`** → returns `CompoundFilter(op="$and", children=[...])`.
4. **`not_(condition)`** → for a single `ModelFieldOperation`, use MongoDB `$not`
   on the field; for compound conditions, use `$nor`.
5. **Update `ModelFieldOperation.to_mongo_filter()`** to accept `CompoundFilter`
   in the conditions list and recursively build the filter dict.
6. **Update `Select.where()`** type hints to accept `CompoundFilter` as well.
7. **Export** `or_`, `and_`, `not_` from `mongotic/__init__.py`.

### Key design considerations

- Keep `CompoundFilter` composable: `or_()` inside `and_()` inside `or_()` etc.
- Ensure `to_mongo_filter()` handles mixed lists of `ModelFieldOperation` and
  `CompoundFilter` seamlessly.
- Consider a `FilterType = Union[ModelFieldOperation, CompoundFilter]` type alias.

- [x] `or_()` produces correct `$or` MongoDB filter
- [x] `and_()` produces correct `$and` MongoDB filter
- [x] `not_()` produces correct `$not` / `$nor` filter
- [x] Nested compositions work (e.g. `or_` inside `and_`)
- [x] Existing implicit-AND `.where()` behaviour is unchanged
