# MGT-014: Add generics to ScalarResult and Select for IDE type inference

## Description

Currently `ScalarResult` and `Select` are non-generic classes. All result methods
(`first()`, `all()`, `one()`, `one_or_none()`) return `MongoBaseModel | None` or
`List[MongoBaseModel]`, regardless of which concrete model was passed to `select()`.

This means IDEs report:

```python
user = session.scalars(select(User).where(User.email == "allen@example.com")).first()
# (variable) user: MongoBaseModel | None   ← wrong
```

Expected:

```python
user = session.scalars(select(User).where(User.email == "allen@example.com")).first()
# (variable) user: User | None             ← correct
```

Same issue affects:
- `ScalarResult.all()` → should be `List[User]` not `List[MongoBaseModel]`
- `ScalarResult.one()` → should be `User` not `MongoBaseModel`
- `ScalarResult.one_or_none()` → should be `User | None`
- `Session.get(User, id)` → should be `User | None` not `MongoBaseModel | None`

## Root Cause

`query.py` — `Select` is a plain class; `select()` returns `Select` (not `Select[T]`).

`orm.py` — `ScalarResult` is a plain class; all result methods are annotated with
the base class `MongoBaseModel` instead of a bound `TypeVar`. `Session.scalars()`
returns `ScalarResult` rather than `ScalarResult[T]`.

## Proposed Fix

### 1. `mongotic/query.py`

```python
from typing import Generic, TypeVar
T = TypeVar("T", bound="MongoBaseModel")

class Select(Generic[T]):
    def __init__(self, orm_model: Type[T]) -> None: ...
    def where(self, ...) -> "Select[T]": ...
    def order_by(self, ...) -> "Select[T]": ...
    def limit(self, value: int) -> "Select[T]": ...
    def offset(self, value: int) -> "Select[T]": ...

def select(orm_model: Type[T]) -> Select[T]: ...
```

### 2. `mongotic/orm.py`

```python
from typing import Generic, TypeVar
T = TypeVar("T", bound="MongoBaseModel")

class ScalarResult(Generic[T]):
    def all(self) -> List[T]: ...
    def first(self) -> Optional[T]: ...
    def one(self) -> T: ...
    def one_or_none(self) -> Optional[T]: ...
    # count() and exists() remain non-generic (return int / bool)
```

`Session.scalars()` and `Session.get()` in both the Protocol and `_Session`:

```python
def scalars(self, stmt: Select[T]) -> ScalarResult[T]: ...
def get(self, model: Type[T], id: Text) -> Optional[T]: ...
```

### 3. Runtime impact

`Generic[T]` is erased at runtime — no behaviour change. Only typing metadata is
affected. No migration needed for users.

## Acceptance Criteria

- [ ] `Select` is `Generic[T]` and `select(User)` is inferred as `Select[User]` by pyright/mypy
- [ ] `ScalarResult` is `Generic[T]` and `session.scalars(select(User)...)` returns `ScalarResult[User]`
- [ ] `.first()` inferred as `User | None`, `.all()` as `List[User]`, `.one()` as `User`
- [ ] `Session.get(User, id)` inferred as `User | None`
- [ ] `Session` Protocol updated to reflect generic signatures
- [ ] All existing tests continue to pass
- [ ] No runtime behaviour change

## Notes

- Use `TypeVar("T", bound=MongoBaseModel)` (not `bound="MongoBaseModel"` string) where possible to avoid forward-ref issues; adjust if circular import prevents it.
- `_hydrate()` in `ScalarResult` is a private method; its annotation can stay as `MongoBaseModel` internally since the cast is safe.
- Consider adding `# type: ignore` or `cast()` inside `_hydrate` if pyright complains about the covariant return.
