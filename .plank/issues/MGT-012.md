# MGT-012: Fix existing bugs in ORM

## Description

Fix three confirmed bugs and one design footgun found in the current `orm.py`
before or alongside the SA v2 refactor.

## Bug 1 — _commit ignores its own parameter (orm.py:221)

`_commit()` receives `update_instances` as a parameter but iterates over
`self._update_instances` instead. The parameter is silently ignored.

```python
# current (wrong)
def _commit(self, ..., update_instances):
    for _update_instance in self._update_instances:  # bug: uses self.
```

Fix: use the passed `update_instances` parameter, or remove the parameter
and always use `self._update_instances` consistently.

## Bug 2 — delete_one runs outside transaction (orm.py:232)

`delete_one` is missing `session=pymongo_client_session`, so it runs outside
the MongoDB transaction:

```python
# current (wrong)
_col.delete_one({"_id": ObjectId(_delete_instance._id)})

# fix
_col.delete_one({"_id": ObjectId(_delete_instance._id)}, session=pymongo_client_session)
```

## Bug 3 — __setattr__ accumulates duplicate updates

Every `__setattr__` call on a model field appends to `_update_instances`
unconditionally. Setting the same field twice queues two updates for the same
field — only the last matters, but both are sent to MongoDB.

Fix: deduplicate by `(instance, field_name)` key before executing, or
maintain a dict `{(instance_id, field): value}` instead of a list.

## Design Fix — Remove default _limit = 5

`QuerySet._limit = 5` silently truncates results. This is a footgun with no
SA equivalent. Remove the default; `None` should mean no limit.

## Acceptance Criteria

- [x] `_commit` parameter usage is consistent
- [x] Delete ops no longer run outside of intended write path (transaction support removed; each write is document-atomic)
- [x] Duplicate field updates deduplicated before flush
- [x] No default limit on queries
